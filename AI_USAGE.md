# AI Tool Usage

This document honestly describes how AI tools were used during this challenge. Fontanella explicitly values AI-assisted development — this reflects how I actually work, not a curated version of it.

---

## Tools used

- **Claude (Anthropic)** — primary pair-programming tool throughout the challenge
- **WebStorm** — IDE with AI-assisted (Claude, Junie) editing for boilerplate-heavy files

---

## What AI was used for

### Scaffolding and boilerplate

- Initial monorepo structure and pnpm workspace configuration
- Prisma schema skeleton (field names and types were then refined manually)
- Express middleware setup — helmet, cors, pino-http wiring
- Docker Compose base configuration
- GitHub Actions workflow structure

### Code generation with review

The following were generated by AI and then reviewed and corrected before committing. In each case the output required meaningful adjustment:

- Repository layer (Prisma query wrappers) — generated, then reviewed for correct index usage, null handling, and missing `include` clauses
- Zod schema definitions — generated, then reviewed for business correctness (the `refine` for `endsAtUtc > startsAtUtc` was added manually)
- React component structure and React Query integration
- Test file scaffolding — the specific test cases were human-designed; the boilerplate setup code was AI-generated

### Debugging

AI pair-debugging was the highest-value use of the tool during this challenge:

**`date-fns-tz@3` timezone bug** — The `format()` function silently ignores the `timeZone` option when the process runs in UTC. This caused 3 failing tests that passed locally (on a machine running in Argentine timezone) but would have failed in CI. The bug was identified through an AI-assisted diagnostic probe script that reproduced the behavior under both `TZ=UTC` and `TZ=America/Argentina/Buenos_Aires`. The fix — pre-converting with `toZonedTime()` before formatting — came from understanding the root cause, not from a suggested code snippet.

**ESM barrel file ambiguity** — `clock/index.ts` re-exporting from `./clock` caused Node's ESM resolver to become confused between the directory and the file. AI helped diagnose the resolution order; the fix (merging the two files) was a structural decision.

**Express 5 `req.params` typing** — Express 5 changed `req.params` values from `string` to `string | string[]`. This surfaced only in CI where strict Prisma types are generated fresh. AI identified the root cause from the TypeScript error; the `param()` helper was the human-authored solution.

## What AI was NOT used for

### Domain modeling and architecture decisions

The following were reasoned through without AI assistance:

- The **timezone snapshot pattern** — capturing timezone at booking time, not at render time. This was the most important domain modeling decision in the system.
- **Half-open interval semantics** for conflict detection — the choice of `<`/`>` over `<=`/`>=`, and why back-to-back appointments should be legal.
- The **Clock abstraction** — the interface design, the three implementations (System, Fixed, Advanceable), and the rationale connecting it to `lcobucci/clock` in PHP.
- The **composition root pattern** — the decision to wire dependencies explicitly in `container.ts` rather than using a DI framework.
- The **`moduleResolution: bundler`** decision — switching from `NodeNext` (which requires `.js` extensions on all relative imports) to `bundler` (which matches what `tsx` and Vite expect).

### Test strategy

- Which scenarios to test and why — the overlap geometry suite, the DST cases, the reschedule exclusion test
- The **fake repository pattern** — hand-written, no mocking library, same interface as real repositories
- The decision to **pin TZ=UTC** in `vitest.config.ts` — this came from understanding why the `date-fns-tz` tests were machine-dependent, not from a suggestion

### Architecture Decision Records

All five ADRs were written manually. AI tools were used to review draft text for clarity, but every decision, tradeoff, and consequence was the result of actual reasoning about the domain.

---

## How I validate AI-generated code

1. **Read every line before committing** — no blind copy-paste
2. **Run the full test suite** after any AI-suggested change
3. **Typecheck with strict TypeScript** — `pnpm -r run typecheck`
4. **If I can't explain a line, I don't ship it** — this caught several cases where generated code was syntactically correct but semantically wrong

## Reflection

The most valuable AI use in this challenge was **pair debugging** — specifically the `date-fns-tz` timezone issue. Without a fast feedback loop for reproducing and diagnosing environment-dependent bugs, this would have cost several hours. The AI-assisted probe script made a subtle bug visible in minutes.

The least valuable AI use was initial scaffolding — structure I would have set up manually anyway, marginally faster.

The areas where AI assistance would have been harmful — domain modeling, test strategy, ADRs — are exactly the areas where the system's correctness and defensibility in a technical review depend on genuine understanding. A half-open interval conflict detection algorithm I didn't reason through myself is a bug waiting to happen. A timezone snapshot pattern I didn't understand would fail in an edge case I didn't anticipate.

The boundary between "AI as a tool" and "AI as a substitute for thinking" is the line I'm trying to keep visible in this document.
